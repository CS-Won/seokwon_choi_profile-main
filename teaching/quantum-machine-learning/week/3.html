Week 3 · Data encoding

Spring 2025 · QIYA — Lecturer: Seokwon Choi

How does classical data enter a quantum computer?

Before the ansatz can do anything useful, the data x∈Rd has to live in a quantum state. This step is called data encoding, and the seminar deck highlights the three classic recipes. Different encodings change the qubit count, the circuit depth, and even what kind of function the model can learn (Schuld & Petruccione, 2018).

1. Basis encoding

The most literal mapping. If the data is a bit string x=(x1​,…,xn​)∈{0,1}n, just place it in the computational basis:

Ebasis​(x)=∣x1​x2​⋯xn​⟩.

Implementation is trivial — apply an X gate on qubit i exactly when xi​=1.

  • Qubits: n (one per bit).
  • Depth: O(1) (parallel X gates).
  • Pros: exact, easy to reason about, plays well with arithmetic oracles.
  • Cons: only handles discrete data; nothing "quantum" happens by default — the state is still classical-looking.

2. Amplitude encoding

Normalize the data and put it into the amplitudes:

Eamp​(x)=∥x∥2​1​j=0∑N−1​xj​∣j⟩,N=2n.

A length-N vector now fits in just n=log2​N qubits — this is the eye-catching exponential compression often mentioned in QML talks.

  • Qubits: log2​N.
  • Depth: a generic state preparation costs O(N) two-qubit gates (Schuld & Petruccione, 2018). So you saved on qubits but spent on gates.
  • Pros: tiny qubit count; matches the "feature vector" mental model in linear ML.
  • Cons: deep preparation circuits unless the input has structure; the global normalization throws away the original norm of the data.

3. Angle encoding

Spend one qubit per feature, each starting in ∣0⟩ and rotated by the corresponding feature:

Eangle​(x)=i=1⨂d​Ry​(xi​)∣0⟩=i=1⨂d​(cos2xi​​∣0⟩+sin2xi​​∣1⟩).

Compared to amplitude encoding, this is the "cheap on the device" option: a single layer of independent single-qubit rotations.

  • Qubits: d (one per feature).
  • Depth: O(1) for the basic version, plus optional entangling layers if you want correlations.
  • Pros: by far the most common choice on NISQ hardware; pairs naturally with the data re-uploading trick we will see in week 5.
  • Cons: the feature dimension is bounded by the qubit count of your device.

Why the encoding choice is a model choice

It is tempting to think of encoding as a separate "data prep" step, like normalizing features in classical ML. It is not. The encoding decides what the final function fθ​(x) can look like.

A useful way to see this (Schuld & Petruccione, 2018; Cerezo et al., 2021): the expectation-value output of a VQA with angle encoding can always be written as a truncated Fourier series in x,

fθ​(x)=ω∈Ω∑​cω​(θ)eiω⋅x.

The set of frequencies Ω is fixed by the encoding (how often, and with what gates, the data enters the circuit), while the coefficients cω​(θ) are what training adjusts. So if your encoder only puts a single rotation per feature, your model can only realize the smallest frequency content — no matter how deep U(θ) is.

This observation is exactly what motivates data re-uploading in week 5: by interleaving encoding and trainable layers we add new frequencies to Ω and get a strictly more expressive model.

Rule of thumb for picking an encoder

  • Discrete data, oracles in the loss: use basis encoding.
  • Dense vectors, lots of features, you have a fast state preparation routine: consider amplitude encoding.
  • Anything else on real NISQ hardware right now: angle encoding with optional data re-uploading is the safe default.
